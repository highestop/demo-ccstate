# Cursor AI 开发规则

## 🎯 核心开发原则

### 代码格式

- 参考项目中的 `.editorconfig` 和 `.prettierrc.json` 配置
- 如修改了配置文件，请运行 `pnpm run validate-config` 进行验证一致性
- 使用 `pnpm run format` 进行格式化
- 使用 `pnpm run format:check` 进行检查

### 命名规范

- 变量和函数：使用 camelCase
- 常量：使用 UPPER_SNAKE_CASE
- 类型和接口：使用 PascalCase
- 文件名：使用 kebab-case
- 目录名：使用 kebab-case

### 编码规范

- 类型优先：优先使用 TypeScript 的类型系统，避免使用 `any`
- 函数式编程：优先使用纯函数和不可变数据结构
- 明确的类型定义：为复杂对象和函数参数定义明确的类型接口
- 避免副作用：函数应该尽可能纯净，副作用应该被明确标识

### 导入导出规范

- 不使用默认导出
- 不使用 `index.ts` 文件作统一导出
- 导入时注意区分是否为只导入类型

### 代码组织

- 单一职责：每个函数和模块应该只有一个明确的职责
- 错误处理：使用明确的错误处理策略，避免静默失败
- 文档注释：为复杂的业务逻辑添加 JSDoc 注释

### 文件组织

- 所有文件平铺在 `src` 目录下，文件名需要遵循命名规范，且清晰表达此文件提供的内容
- 每个源码文件配有一个测试文件，测试文件名与源码文件相同，后缀为 `.test.ts`
- 测试文件与源码文件放在同一目录下

### 提交规范

- 无需创建分支，直接在 `main` 分支上创建 Commit
- 参考 Conventional Commits 格式和项目中的 `commitlint.config.js` 配置
- 使用 `pnpm run commitlint` 进行提交信息检查
- 提交前运行 `pnpm run precommit`

## 🛠️ 开发工作流

### 开发工具

- `Pnpm`: 包管理工具
- `Prettier`: 代码格式化
- `ESLint`: 代码质量检查
- `OxLint`: 快速代码检查
- `Lefthook`: Git hooks 支持
- `Commitlint`: 提交信息规范

### 开发命令

```bash
# 代码格式化
pnpm run format # 格式化所有文件
pnpm run format:check # 检查格式化状态

# 代码质量检查
pnpm run lint # 运行 eslint
pnpm run lint:fix # 自动修复 eslint 问题

pnpm run oxlint # 运行 oxlint 快速检查
pnpm run oxlint:fix # 自动修复 oslint 错误

# 类型检查
pnpm run type-check # TypeScript 类型检查

# 配置验证
pnpm run validate-config # 验证配置一致性

# 完整检查
pnpm run precommit # 运行所有检查
```

## 📋 文档维护规则

### 文档更新原则

- `README.md` 不可修改，保持项目概述稳定
- 不要在根目录下创建其它 md 文档，只对 `.CLAUDE.md` 文件进行更新
- 确保最新文档内容与项目中的其它内容一致，如不一致也请一并更新
- 保持文档结构清晰，使用合适的标题层级
- 必要时提供具体的代码示例和命令
- 对 `.CLAUDE.md` 文件更新时，须寻求用户同意
